### 4. ✅ Capacity Analysis

**Metrics Calculated:**
- **Raw Capacity** (TB): Width × Drives/Node × Drive Capacity
- **Usable Capacity** (TB): Raw × (Threshold / Width)
- **Storage Efficiency** (%): (Threshold / Width) × 100
- **Expansion Factor** (×): Width / Threshold

**Formula Example (9-6-7):**
```
Raw = 9 × 12 × 8 = 864 TB
Usable = 864 × (6/9) = 576 TB
Efficiency = (6/9) × 100 = 66.7%
Expansion = 9/6 = 1.5×
```

**UI Features:**

### 5. ✅ Failure Tolerance Analysis

**Metrics Calculated:**
- **Node Read Tolerance**: Width - Threshold
- **Node Write Tolerance**: Width - Write Threshold
- **Site Tolerance**: 0 or 1 (based on topology)
- **Drive Tolerance**: Node Tolerance × Drives/Node

**Example (9-6-7, Single-Site):**
```
Read: 9 - 6 = 3 nodes
Write: 9 - 7 = 2 nodes
Site: 0 (single-site)
Drives: 3 × 12 = 36 drives
```

### 6. ✅ Availability & Durability

**Metrics Calculated:**
- **Durability (9's)**: 9 + (Width - Threshold)
- **Durability Percentage**: 99.9999999999% format
- **MTTDL (Years)**: Simplified exponential model

**Example (9-6-7, 0.5% AFR):**
```
Durability: 9 + 3 = 12 nines
Percentage: 99.9999999999%
MTTDL: ~1.85 million years
```

**Formula Logic:**
```javascript
Total Drives = Width × Drives/Node
MTBF = 8760 hours / (AFR / 100)
System MTTFF = MTBF / Total Drives
MTTDL ≈ (MTTFF / 8760) × 100^(Node Tolerance)
```

**UI Features:**
- Two-metric grid (durability & MTTDL)
- Large cyan values with text shadow glow
- Explanation includes site topology impact
- Correlated failure warnings for single-site

**Note:** MTTDL is **simplified** for educational purposes. Production use requires combinatorial probability models.

### 7. ✅ Performance Indicators

**Metrics Calculated:**
- **Write Amplification** (×): Width / Threshold
- **Rebuild Time Estimate**: Based on drives, capacity, 100 MB/s rate
- **Network Overhead** (%): ((Width - Threshold) / Threshold) × 100

**Example (9-6-7, 12×8TB):**
```
Write Amp: 9/6 = 1.5× (1 TB → 1.5 TB physical)
Rebuild: (12 × 8 TB × 1048576 MB) / (100 MB/s × 3600 s/hr) 
       = 279 hours (11.6 days)
Network: ((9-6)/6) × 100 = 50%
```

**UI Features:**

## 🔑 Key Technical Decisions

### 1. ✅ Single-File Architecture
**Decision:** Embed CSS + JS in HTML (no separate files)

**Rationale:**
- Maximum portability (one file to share)
- No file path issues
- Works locally without server
- Easy to email, download, distribute

**Trade-off:** Harder to edit with syntax highlighting (but worth it for portability)

---

### 2. ✅ Vanilla JavaScript (Zero Dependencies)
**Decision:** No frameworks (React, Vue), no libraries (jQuery, Lodash)

**Rationale:**
- No build process required
- No npm, webpack, babel, etc.
- Instant load time (no bundle parsing)
- Educational codebase (easy to understand)

**Trade-off:** More verbose code (but cleaner for learning)

---

### 3. ✅ IBM Carbon Design Inspiration
**Decision:** IBM blue/cyan palette, IBM Plex Sans font

**Rationale:**
- Aligns with IBM COS branding
- Professional, enterprise aesthetic
- High contrast (accessibility)
- Official IBM typeface (via Google Fonts CDN)

**Trade-off:** Requires internet for font (but graceful fallback)

---

### 4. ✅ Simplified MTTDL Calculation
**Decision:** Use exponential approximation (not combinatorial model)

**Rationale:**
- Educational tool, not production planner
- Complex binomial models obscure learning
- Order-of-magnitude accuracy sufficient
- Real deployments use vendor tools anyway

**Trade-off:** Not suitable for official capacity planning (noted in docs)

---

### 5. ✅ Rebuild Threshold = Threshold
**Decision:** Auto-calculate, don't expose as separate input

**Rationale:**
- IBM COS uses same value (Threshold) for rebuilds
- Reduces user confusion
- Simplifies UI (fewer inputs)
- Can expose later if other vendors need it (MinIO)

**Trade-off:** Less flexible (but correct for IBM COS)

---

### 6. ✅ Hard-Coded Single/3-Site (Not 2-Site)
**Decision:** Only allow single or 3-site configurations

**Rationale:**
- 2-site has split-brain risks (Reed-Solomon)
- Educational focus: Show correct architectures only
- Cleversafe best practices: 1-site or 3-site (not 2)
- Prevents users from configuring invalid topologies

**Trade-off:** Less flexible (but prevents errors)

---

### 7. ✅ Collapsible AFR Section (Not Always Visible)
**Decision:** Hide AFR by default, expand on click

**Rationale:**
- Saves screen real estate
- Most users don't adjust AFR (0.5% is standard)
- Advanced users know where to find it
- Keeps main UI clean

**Trade-off:** One extra click for AFR changes (acceptable)

## 📝 Known Issues & Limitations

### Documented Limitations

**MTTDL Simplification:**
- ⚠️ Uses exponential approximation (not combinatorial)
- ⚠️ Does NOT account for rebuild time windows
- ⚠️ Does NOT model correlated failures
- ✅ Clearly noted in UI and README

**Site Tolerance Simplification:**
- ⚠️ Assumes even node distribution (3 per site for 9 nodes)
- ⚠️ Does NOT handle asymmetric configs (4-3-2 split)
- ✅ Simplified for educational clarity

**Drive Tolerance:**
- ⚠️ Assumes entire node failure (all drives)
- ⚠️ Does NOT model individual drive failures within operational node
- ✅ IBM COS actual behavior more nuanced (noted)
